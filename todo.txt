The design system has a solid start, but it is still missing several foundations and reusable states. Based on current duplication and usage, I’d
  prioritize these additions:

  1. Complete theme foundations

  The current theme mainly configures colors, inputs, dividers, and bottom navigation. It does not configure typography, cards, dialogs, chips,
  switches, buttons, tables, app bars, or navigation rails. The bundled Montserrat font is not actually applied.

  Add:

  - DesignTypography
  - Semantic colors such as success, warning, info, disabled, and destructive
  - Elevation/shadow tokens
  - Component size tokens
  - Breakpoints and content-width tokens
  - Motion duration and curve tokens
  - Complete light/dark component themes

  See modules/design_system/lib/src/theme/design_theme.dart:6.

  2. State presentation components

  Feature modules repeatedly construct loading, empty, and error layouts manually. For example, the users screen has three separate inline states.

  Add:

  - BaseLoadingView
  - BaseEmptyState
  - BaseErrorState
  - BaseResultView or BaseAsyncContent
  - Optional skeleton components

  These should remain visual and accept strings, icons, and callbacks rather than depending on Bloc or Failure.

  See modules/admin_users/lib/src/features/users/presentation/widgets/users_list_view.dart:103.

  3. Switch list tile

  This is the clearest immediate extraction. UserPreferencesNotificationSwitchCard is duplicated exactly in admin and client preferences.

  Add a generic:

  BaseSwitchListTile(
    title: ...,
    value: ...,
    loading: ...,
    onChanged: ...,
  )

  See modules/admin_preferences/lib/src/features/user_preferences/presentation/widgets/user_preferences_notification_switch_card.dart:4 and modules/
  client_preferences/lib/src/features/user_preferences/presentation/widgets/user_preferences_notification_switch_card.dart:4.

  4. Search, filters, and status indicators

  The admin users module currently implements detailed styling itself.

  Add:

  - BaseSearchField
  - BaseFilterButton
  - BaseFilterChip
  - BaseStatusBadge
  - BaseVerifiedIndicator
  - BaseDateRangeField

  The design-system components should accept generic labels, colors, icons, and values. Mapping AdminUserStatus to those values must stay inside
  admin_users.

  See modules/admin_users/lib/src/features/users/presentation/widgets/users_search_bar.dart:5 and modules/admin_users/lib/src/features/users/
  presentation/widgets/user_status_card.dart:7.

  5. Responsive layout primitives

  There are several unrelated hardcoded widths such as 172, 280, 400, 560, and 600.

  Add:

  - DesignBreakpoints
  - BaseContentContainer
  - BaseResponsivePage
  - BaseResponsiveDialog
  - Possibly BaseSidebarLayout

  This would standardize mobile, tablet, and admin-web behavior without moving app navigation ownership into the design system.

  6. Data presentation

  If more admin modules will contain management screens, add:

  - BaseDataTable
  - BaseTableContainer
  - BasePagination
  - BaseTableToolbar
  - Column header and sortable-header helpers

  Keep row models, navigation, filtering logic, and localization in the feature package.

  7. Swipe/action list tile

  Admin and client profile contain essentially the same slidable session item. A generic BaseActionListTile or BaseSlidableListTile could own border
  radii, action colors, and layout while callers supply action labels and callbacks.

  This is useful, but lower priority because it adds flutter_slidable to the design-system dependency surface.

  8. Component documentation and tests

  The package currently has only four tests for the whole public component surface. Before adding many components, it needs:

  - A component gallery or Widgetbook
  - Light/dark visual coverage
  - Golden tests
  - Text-scaling and overflow tests
  - Accessibility/semantics tests
  - Keyboard and focus tests for web
  - RTL and all three locale checks

  I would implement the first batch in this order:

  1. Theme and semantic tokens
  2. BaseEmptyState, BaseLoadingView, BaseErrorState
  3. BaseSwitchListTile
  4. BaseSearchField, BaseFilterChip, BaseStatusBadge
  5. Breakpoints and responsive containers
  6. Component gallery and golden/accessibility tests

  One cleanup should accompany this: remove the temporary AppSpacing/AppRadii aliases and consistently use DesignSpacing/DesignRadii; also normalize
  names like DesignCard and BottomSheetList to the established Base* convention.
